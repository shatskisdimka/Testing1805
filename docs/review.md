# Заметки по React Native (тестовое задание)

## Почему Expo Go "не видел" сервер (VPN)

Телефон не подключался к `expo start`, хотя был в одной Wi-Fi сети с компом.
Причина — на компе одновременно был активен VPN-адаптер (AmneziaVPN). В обычном
LAN-режиме Expo CLI определяет, какой IP-адрес компа вписать в QR-код/URL для
подключения, и при активном VPN он может взять адрес VPN-интерфейса
(`100.92.209.x`) вместо реального адреса локальной сети (`192.168.1.x`). Телефон
физически не может достучаться до VPN-only адреса — отсюда "сервер не виден".

**Решение:** запускать Expo через туннель — `expo start --tunnel` (использует
`ngrok`). Туннель публикует дев-сервер через внешний адрес ngrok, который
телефон видит независимо от VPN/LAN-роутинга на компе. Минус — медленнее, чем
обычный LAN-режим, это нормально для разработки.

## Предупреждения о несовместимых версиях пакетов

При `npx expo start` Expo CLI пишет предупреждения вида:
```
expo-status-bar@2.0.1 - expected version: ~3.0.9
react-native@0.81.4 - expected version: 0.81.5
```
Это значит, что версии конкретных пакетов в `package.json` не совпадают с тем,
что ожидает установленная версия `expo` (SDK). Несовпадение версий — частая
причина нестабильной работы/крашей (именно так и было: в репозитории остались
версии пакетов от попытки на Expo SDK 55, а сам `expo` в какой-то момент
откатили на SDK 54).

**Чинится командой:** `npx expo install --fix` — подтягивает версии пакетов под
ту, что ожидает установленный `expo`.

## Типизация (TypeScript): убираем `any`

Задача — избавиться от `any` в пропсах, данных API и хуках, и разнести типы по
отдельным файлам вместо одного "мусорного" `types/index.ts`.

- Завели `types/news.ts` (`NewsItem`, `NewsResponse`) и `types/order.ts`
  (`OrderItem`), а `types/index.ts` сделали просто barrel-файлом
  (`export * from './news'` и т.д.) — типы по смыслу, не всё в одной куче.
- `fetchNews()` явно типизирован как `Promise<NewsItem[]>`, и у `response.json()`
  тоже указан тип — без этого TS считает результат `any`, и тип "утекает" дальше
  по коду везде, где используются данные.
- В компоненте `useState<NewsItem[]>([])` и `useState<boolean>(true)` вместо
  `useState([])` — иначе TS выводит тип состояния как `never[]`/неявный `any[]`,
  и при дальнейшей работе с массивом не будет подсказок и проверок.
- Заодно нашли реальный баг: `mockOrders.ts` использовал тип `OrderItem`, который
  нигде не был объявлен/импортирован — без строгой типизации этот файл вообще не
  компилировался.

## Обработка ошибок API: сеть vs HTTP-статус

Это два разных по природе сбоя, и важно не путать их в одном `catch`:

1. **Сетевая ошибка** — сам `fetch()` бросает исключение (нет интернета, сервер
   недоступен, DNS не резолвится и т.д.). Ловится обычным `try/catch` вокруг
   вызова `fetch`.
2. **HTTP-ошибка** — `fetch()` успешно отработал, пришёл ответ, но статус не
   2xx (404, 500 и т.д.). `fetch` в этом случае НЕ бросает исключение сам —
   нужно явно проверять `response.ok` и бросать ошибку вручную.

Сделали свой класс `ApiError extends Error` с необязательным полем `status`,
чтобы экран мог показать осмысленный текст в обоих случаях (и опционально знать
код, если он есть). На экране — три состояния рендера: `loading` / `error` (текст
+ кнопка "Повторить") / список с данными.

**Как проверяли:** для сетевой ошибки нужен реальный обрыв соединения (авиарежим
не сработал на деле). Поэтому тестировали через код — временно меняли URL на
несуществующий путь (`/postss`), чтобы получить гарантированный `404` и увидеть
ветку `!response.ok`, затем возвращали URL обратно.

## React.memo и список (FlatList)

Когда заменили `ScrollView` на `FlatList` для списка новостей, карточку вынесли в
отдельный компонент `NewsListItem` и обернули в `React.memo`.

**Что это даёт:** `React.memo` пропускает повторный рендер компонента, если его
пропсы не изменились (сравнение по ссылке/значению). Для списка это значит — если
родительский экран перерисовался из-за какого-то *своего* состояния (не связанного
с данными конкретной карточки), сами карточки рендериться заново не будут.

**Проверили на деле:** добавили временный `console.log` в `NewsListItem`,
перезагрузили экран — в логе ровно 100 строк (по числу новостей с
jsonplaceholder — это фиксированный мок-датасет, всегда ровно 100 постов),
без повторов и пропусков. Это подтвердило, что `FlatList` + `keyExtractor`
(по `item.id`) рендерят список корректно — каждый элемент один раз.

**Важный нюанс:** этот тест проверил именно *корректность* рендера списка, а не
сам эффект экономии от `memo`. Эффект `memo` проявляется только когда родитель
перерисовывается, а проп `item` остаётся тем же самым (по ссылке). У нас
единственный триггер ре-рендера — `loadNews()`, который каждый раз делает новый
fetch и создаёт новый массив `data` с новыми объектами — то есть `item` каждый раз
новый, и `memo` тут законно ничего не блокирует (рендер всех карточек заново —
это правильно, не лишний). Реальная польза от `memo` появится, если в экране
заведётся ещё какое-то состояние не про данные (например, поиск/фильтр/тема) —
тогда карточки не будут зря перерисовываться при его изменении.

**`useCallback` для `renderItem`/`keyExtractor`:** обернул их в `useCallback` с
пустым массивом зависимостей, чтобы при каждом рендере экрана `FlatList` получал
ту же самую функцию (по ссылке), а не новую — иначе `FlatList` (и `memo` внутри
карточек) видели бы "новый" проп-функцию каждый раз и оптимизация не работала бы.

## Анимация появления карточек (react-native-reanimated)

Для плавного появления карточек в списке достаточно обернуть содержимое
`NewsListItem` в `Animated.View` из `react-native-reanimated` и задать пропс
`entering={...}` — готовый layout-animation пресет библиотеки, без написания
своих воркле́т-функций. Перебрали несколько пресетов (`FadeInDown`, `SlideInDown`)
и в итоге остановились на `SlideInLeft.duration(400)` — карточка въезжает слева,
эффект простой и легко объяснить.

**Почему анимация сначала "не было видно":** `FlatList` по умолчанию домонтирует
карточки сильно заранее (большой `windowSize`), то есть карточка уже отрисована
и анимация уже отыграла до того, как до неё реально доскроллили — выглядело как
будто анимации нет. Решилось сужением окна предзагрузки:
`initialNumToRender={6}`, `maxToRenderPerBatch={6}`, `windowSize={2}`,
`removeClippedSubviews` — карточки монтируются ближе к моменту, когда они показываются
на экране, и `entering`-анимация совпадает по времени со скроллом к ним.

**Почему анимация играла только один раз (промежуточная гипотеза):** `jsonplaceholder`
отдаёт одни и те же 100 постов с одними и теми же `id` при каждом фетче. Раз
`keyExtractor` строился по `item.id`, при повторной загрузке (например, кнопка
"Повторить") React видел те же ключи и просто переиспользовал уже смонтированные
компоненты карточек — `entering` срабатывает только при настоящем монтировании,
повторно не играл. Пробовали чинить, привязав `keyExtractor` к счётчику версии
загрузки (`` `${loadVersion}-${item.id}` ``), но в итоге этот вариант откатили —
не дал того результата, который реально хотелось (карточки должны выезжать по
мере скролла, а не пересоздаваться целиком при рефетче). **В текущем коде этой
версии нет** — `keyExtractor` обычный, по `item.id`. Финальное решение — именно
сужение `windowSize` (см. выше): раз карточки монтируются непосредственно перед
тем, как попадают в видимую область, анимация естественным образом проигрывается
при каждом скролле к новому элементу, и проблема "играет один раз" перестала быть
актуальной для реального сценария использования.

**Babel-плагин не нужен руками:** показалось логичным, что для reanimated
обязательно нужен `babel.config.js` с плагином `react-native-reanimated/plugin`
(так было в старых версиях/во многих гайдах). Но в установленном
`babel-preset-expo` (том самом, который Expo подключает по умолчанию без
какого-либо `babel.config.js` в проекте) уже встроена автоматическая проверка:
если `react-native-reanimated` найден в `node_modules`, плагин подключается сам
(см. `node_modules/expo/node_modules/babel-preset-expo/build/index.js`, строка
~284). Поэтому создавать свой `babel.config.js` не потребовалось — он был бы
лишним и даже мог сломать резолв пакетов (`babel-preset-expo` лежит вложенно в
`node_modules/expo/node_modules/...`, и не резолвится из кода в корне проекта).

**Реальный краш после этого:** `Exception in HostFunction: <unknown>` при старте
(плюс смежные ошибки LogBox/React version mismatch). Причина — несовпадение
версий по двум направлениям:

1. `react`/`react-native`/`expo-status-bar`/`react-native-reanimated` в
   `package.json` были не те, что ожидает установленный Expo SDK 54
   (`npx expo start` об этом честно предупреждал ещё раньше, но было
   проигнорировано до первого реального краша). Починили: `npx expo install --fix`.
2. Более тонкий случай — `react-native-worklets` (нативная зависимость
   `reanimated`) npm подтянул как пир-зависимость в последней подходящей по
   диапазону версии `0.8.3`, а в `node_modules/expo/bundledNativeModules.json`
   (таблица того, что реально зашито нативно в Expo Go для SDK 54) написано
   `0.5.1`. JS-сторона дёргала нативную функцию по протоколу 0.8.x, а в Expo Go
   натив был от 0.5.1 — несовпадение ABI и есть тот самый `HostFunction`-краш.
   Починили явным пином: `npx expo install react-native-worklets@0.5.1`.

**Вывод:** при использовании библиотек с нативным кодом в связке с Expo Go важно
сверяться не только с `peerDependencies` (там обычно widened-диапазон, который
просто не ломает сборку), а с `bundledNativeModules.json` внутри `expo` —
именно там зафиксированы версии, которые реально совместимы с конкретным
бинарником Expo Go.

## Часть 2 — экран «Заказы» и навигация (2.1, 2.2, 2.4)

Сделали новый экран с нуля по макету и навигацию между ним и экраном новостей
из Части 1.

**Стилизация — остались на `StyleSheet`, не стали тащить NativeWind**, даже
несмотря на то, что в пет-проектах привычнее писать на Tailwind-классах.
Причина — консистентность: весь остальной код в проекте уже на `StyleSheet`, а
смешивать два подхода к стилям в одном небольшом проекте выглядело бы как
несогласованность, а не осознанный выбор инструмента. Плюс NativeWind тащит свой
babel-плагин/конфиг — ещё один потенциальный источник версионных проблем (а мы
их уже наелись с reanimated/worklets).

**Иконки — `@expo/vector-icons`** (`Ionicons`), официальный пакет Expo, ставится
через `expo install` под версию из `bundledNativeModules.json`, как и остальные
нативные зависимости. Полный список доступных иконок и их `name` — на
https://icons.expo.fyi/.

**Навигация — `@react-navigation/native` + `@react-navigation/bottom-tabs`**,
а не `expo-router` с файловым роутингом (`app/`, `_layout.tsx`, `(tabs)`).
Решили так, потому что:
- `expo-router` меняет entry point всего проекта (`"main": "expo-router/entry"`
  вместо `index.ts`) и требует структуру папок `app/` — для двух экранов это
  перестройка всего скелета, а не локальное добавление фичи.
- В проекте и так всё устроено явно (`App.tsx` руками компонует, что рендерить) —
  классический `@react-navigation` ближе к этому стилю, чем файловая магия.
- Меньше новых нативных зависимостей = меньше риска словить очередной
  version-mismatch (как было с `react-native-worklets`).

Технически: `@react-navigation/native`/`bottom-tabs` тянут за собой
`react-native-screens` и `react-native-safe-area-context` как peer-зависимости —
поставили их тоже через `expo install` (важно — именно через него, а не голый
`npm install`, чтобы версии совпали с тем, что зашито в Expo Go, см. историю с
worklets выше).

**Safe area переехала из `App.tsx` в сами экраны.** Раньше `App.tsx` оборачивал
единственный экран в `SafeAreaView` из core `react-native` (слабая реализация,
плохо работает с отступами на Android). Теперь `App.tsx` рендерит
`SafeAreaProvider` + `NavigationContainer`, а отступы под статус-бар/нотч каждый
экран обрабатывает сам через `react-native-safe-area-context`:
- `LegacyListScreen` — обёрнут в `SafeAreaView` (из `safe-area-context`,
  `edges={['top']}`).
- `OrdersScreen` — синий хедер специально НЕ обёрнут в `SafeAreaView`, чтобы
  цвет фона уходил под сам статус-бар (как в макете), а внутренний контент
  (заголовок/поиск) сдвинут вручную через `useSafeAreaInsets().top`.
- Нижний отступ (home indicator на iOS) для таб-бара `bottom-tabs` обрабатывает
  сам, автоматически, через тот же контекст — отдельно делать ничего не пришлось.

**Моки заказов:** изначально все 8 записей в `mockOrders.ts` были идентичны
(один и тот же номер/сумма/статус). Подправили вручную под более реалистичный
вид для превью — разные номера, суммы, кол-во товаров, статусы (Выполнен / В
обработке / Отменён) и даты в формате `ДД.ММ.2026`, как в макете.

## Часть 2.3 — управление состоянием (Zustand)

Для экрана «Заказы» выбрали Zustand, а не TanStack Query/Context/просто
`useState`. Рассуждение: у заказов нет настоящего сервера — данные локальные/
моковые, значит это classic client state, а не server state. TanStack Query
создан для кэширования и синхронизации с реальным API (стейл-тайм, рефетчи,
инвалидация), а тут нет такого асинхронного ресурса — притворяться, что он есть,
было бы искусственно. (Для новостей в Части 1, где запрос к настоящему API
действительно есть, наоборот — TanStack Query был бы более уместен, чем ручной
`useState`/`useEffect`, который мы там написали; но это уже сделано и
переписывать не стали.)

**Стор:** `store/useOrdersStore.ts` — `orders`, `searchQuery` и три действия:
`setSearchQuery`, `refresh` (сброс к исходным моковым данным и пустому поиску),
`addOrder` (добавляет мок-заказ в начало списка). Поиск в `OrdersScreen`
фильтрует список по номеру заказа через `useMemo`, кнопки "Обновить"/"+ Новый
заказ" дёргают действия стора напрямую.

**Баг, который нашли:** в `createNewOrder` следующий номер заказа считался как
`orders[0].number + 1` — то есть от **первого** элемента списка, а не от
**максимального** номера. Моки идут по возрастанию (`12450...12457`), значит
`orders[0]` — самый маленький номер, и новый заказ получал номер, который уже
существует у другого заказа дальше в списке (задвоение). Починили через
`orders.reduce((max, order) => Math.max(max, order.number), 0)`.

**Как проверяли стор без UI:** `zustand`'ом можно пользоваться и вне React —
`zustand/vanilla` экспортирует `createStore`, у которого есть `getState()` и
`setState()`. Прогнали ту же логику стора прямо в Node (`node -e "..."`),
вызвали `addOrder()`/`refresh()` напрямую и посмотрели на `getState().orders` —
получили ожидаемые длину массива и номер нового заказа, без необходимости
гонять весь RN/Expo Go ради проверки чистой бизнес-логики.

## Фикс вкладки «Новости» — зазор перед таб-баром

После добавления навигации заметили, что на вкладке «Новости» внизу экрана,
перед таб-баром, всегда висела тонкая серая полоса — причём не зависящая от
скролла (видна и в середине списка, и в конце). Сначала подумал, что дело в
агрессивной виртуализации `FlatList` (`windowSize`), но это было не так.

**Реальная причина:** у `container` в `LegacyListScreen` было `padding: 16` —
то есть отступ со всех четырёх сторон сразу. Отступ снизу — это фиксированная
часть `View`, она не скроллится вместе с контентом, поэтому полоса висела
поверх таб-бара всегда, на любой позиции скролла. В `OrdersScreen` такой
проблемы не было, потому что там отступы внутренние (`contentContainerStyle`
самого списка), а не на родительском контейнере.

**Фикс:** заменили `padding: 16` на `paddingHorizontal: 16, paddingTop: 16` —
убрали именно нижний отступ у контейнера, ничего больше не трогая (`windowSize`
и остальное осталось как было). Теперь список доскроллливается вплотную к
таб-бару, без мёртвой зоны.
